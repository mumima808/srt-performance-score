# CONCEPT — SRTは、動画のMIDIである

## ひとことで言うと

SRTファイルは字幕ではない。**動画を演奏するための楽譜**である。

---

## フレームの転換

| 古い見方 | 新しい見方 |
|----------|------------|
| SRT = 字幕。編集後に付け加えるもの | SRT = 楽譜。すべてをその上に載せるもの |
| 撮影 → 編集 → 字幕 | SRT（設計） → 映像を載せる |

この転換は、技術の話ではない。**見方の話**だ。

---

## MIDIとの類比

MIDIは音楽制作を変えた。

音の「意味」だけを分離した。音色はシンセが決める。タイミングと音程だけを伝える。だから何十年も、あらゆる機器をつなぐHUBであり続けた。

SRTも同じことができる。

| MIDI | SRT |
|------|-----|
| 音程・タイミング | 言葉・タイムコード |
| 音色はシンセが決める | 映像はDaVinciが決める |
| 声はシンガーが決める | 声はTTSが決める |
| あらゆる機器をつなぐ | あらゆるツールをつなぐ |

そして、MIDIと決定的に違う点がひとつある。

**SRTは、人間が読める。**

MIDIのバイナリは人間には読めない。でもSRTはテキストだ。誰でも書ける。編集できる。GitHubで差分管理できる。

## なぜ気づいたのか

動画編集ツールは、長い間タイムラインの設計から抜け出せていない。

縦軸にトラック、横軸に時間——これは録音スタジオの物理的制約を、そのままデジタルに持ち込んだ設計だ。DaVinci Resolve も、DTM も、DAW も、本質的には同じ思想の上に立っている。

問題はここにある。

**ツールを学ぶことで力が尽きて、創造に到達する前に終わる人が多すぎる。**

Vrewには、「ことばの一塊」という単位がある。静止画と短い動画と音声を、言葉を単位としてパケットに梱包できる。前後に「空白（間）」を置くことができる。

その瞬間に気づいた。

これは時間軸ではなく、**言葉を単位とした時間の制御装置**だ。SRTはその設計図になれる。

## 沈黙を設計する

これは、プロが知っていて、めったに言語化しなかった技術だ。

アイドルのステージ：「Are you ready? — Yeah!」  
漫才師のパンチラインの前の間。  
落語家の「えぇ…」が客を前のめりにさせる。  
ヒップホップのMCが1万人に歌詞の続きを言わせる。

これらはすべて同じ仕組みだ。**発話 → 沈黙 → 応答**。設計された時間の隙間が、観客をパフォーマーに変える。

SRTのタイムコードで言えば——空白のギャップは空っぽではない。それはキューだ。

DJも同じことをしている。フィルターとレイヤーで緊張を高め、クラウドを限界まで引っ張る。そしてドロップ。その安堵は身体的だ。これもタイムコード構造だ。SRTファイルとして書ける。

**DJはずっと、パフォーマンス・スコアを作曲していた。それを「場を読む」と呼んでいただけだ。**

---

## DJコントローラーとパッドサンプラーが近い理由

DTMもDAWも、タイムラインの呪縛から抜け出せていない。

しかしDJコントローラーやパッドサンプラーは違う。パッドは「塊」を叩く。音を時間軸に置くのではなく、意味のある単位をトリガーする。順序も間も、その場で決める。

音楽理論を知らなくても、パフォーマンスができる。

Vrewの「ことばの一塊」も同じ思想だ。編集理論を知らなくても、言葉を並べるだけで時間を設計できる。

## SRTをHUBとした世界

SRTが動画のMIDIになれば、何が起きるか。

```
Audio / Language / Story
  ↓  Vrew
SRT = Performance Score
  ↓  DaVinci Resolve    → 映像編集
  ↓  TTS                → ナレーション生成
  ↓  LLM + AIキャラクター → 演技・パフォーマンス
  ↓  自動動画生成        → ワンソースマルチユース
```

言葉とタイムコードだけを書けば、あとはツールが演奏する。

**これは、YouTube動画をクリエイトするすべての人間とAIに関係がある。**
